# RelayManager 项目健康度优化报告

**日期**: 2026-06-26 | **范围**: `server.js` (2782 行) + `index.html` (2539 行) | **总代码量**: ~5400 行

---

## 一、路由与页面层级（信息架构）

### 1.1 当前架构

项目不是 SPA 框架（无 React/Vue Router），而是**纯 HTML + 服务端路由**。前端通过 7 个 Tab 实现页面切换，后端通过 if-else 链处理 ~45 个 API 路由。

**前端 Tab 结构：**
```
/                        → panel-overview（总览仪表板）
/?tab=claude-code        → panel-claude-code（Claude Code CLI 配置）
/?tab=claude-desktop     → panel-claude-desktop（Claude Desktop 3P 配置）
/?tab=codex-cli          → panel-codex-cli（Codex CLI 配置）
/?tab=codex-desktop      → panel-codex-desktop（Codex Desktop 配置）
/?tab=models             → panel-models（模型映射）
/?tab=gateway            → panel-gateway（中继网关）
```
> **优先级：中**

**问题：**
- **平级过载**：7 个 Tab 全部平级排列，但 `models`（模型映射）和 `gateway`（中继网关）在逻辑上是**跨产品共享的基础设施**，不应该与 4 个产品配置 Tab 处于同一层级。当前信息架构让用户困惑「我是在配置一个产品还是全局设置」。
- **命名歧义**：`models` Tab 同时承载"从上游拉取模型列表"和"为各产品做模型映射"两个不同任务。名称只暗示了前者。
- **Tab 数量超出短期记忆容量**：7 个 Tab 处于 Miller's Law（7±2）的临界点，新增任何功能都会突破可发现性边界。

> **建议：**
> 1. 将 Tab 分成两组：**产品配置**（4 个产品）和**全局设置**（网关、模型映射），用视觉分隔或二级导航区分。代价约 20 行 CSS。
> 2. 将 `models` 重命名为 `模型映射/同步` 或将其功能融合进各产品 Tab 的"模型映射"区域，删除独立 Tab。
> 3. 考虑将 `gateway` Tab 提升为顶层概念（因为它是核心差异化功能），其他产品 Tab 围绕它组织。

### 1.2 服务端路由组织

**当前状态：** 45 个 API 路由全部通过 `if (method === 'GET' && url.pathname === '/api/...')` 链式判断（`server.js` 第 2207-2750 行），无任何路由分组、中间件或命名空间约定。

> **优先级：高**

**具体问题：**

| 路由 | 所属产品 | 前缀不统一？ |
|---|---|---|
| `/api/claude-code/configs` | Claude Code | ✅ 统一 |
| `/api/claude-desktop/config/create` | Claude Desktop | ✅ 统一 |
| `/api/codex-cli/config/list` | Codex CLI | ❌ 多了 `/list` |
| `/api/codex-desktop/config/list` | Codex Desktop | ❌ 多了 `/list` |

Claude 系列使用 `/configs`（复数），Codex 系列使用 `/config/list`（子路径）。命名风格不统一，增加记忆负担。

> **建议：**
> 1. 统一为 `GET /api/{product}/configs` 和 `POST /api/{product}/config/{action}`，消除 Codex 的 `/list` 后缀差异。
> 2. 提取一个 `registerCrudRoutes(prefix, handlers)` 工具函数消除 ~200 行重复路由注册代码（详见第二章）。

## 二、功能重复性检测（DRY）

### 2.1 四产品 CRUD 的「复制粘贴工程」

这是整个项目最显著的重复问题。Claude Code CLI、Claude Desktop 3P、Codex CLI、Codex Desktop 四个产品各自有一套**完全平行的**配置预设管理系统。

**Server.js 端：** 每个产品实现了 8 个函数：读存储、写存储、Sanitize、List、Create、Apply、Update、Delete、Rename、Export。

| 产品 | 函数前缀 | 存储文件 | 与 CC/CX/CXD 共用模式的差异 |
|---|---|---|---|
| Claude Code CLI | `CC` | `cc-configs.json` | — |
| Claude Desktop 3P | `CD` | `_meta.json` + `<id>.json` | 唯一例外（文件系统存储），但接口相同 |
| Codex CLI | `CX` | `codex-cli-configs.json` | 与 CC 100% 同构 |
| Codex Desktop | `CXD` | `codex-desktop-configs.json` | 与 CC 100% 同构 |

**Rename 函数（4 份完全相同的代码）：**
```javascript
// 4 个产品的 Rename 函数，仅 readXxxStore / writeXxxStore 调用不同
function renameCCConfig(id, name) {
  const store = readCCStore();
  const e = store.entries.find(e => e.id === id);
  if (!e) throw new Error('not found');
  e.name = name;
  writeCCStore(store);
}
```

> **优先级：高** | **估计重复行数：server.js ~400 行 / index.html ~550 行**

**Index.html 端：** 每个产品有 7 个 UI 管理函数（load / switch / new / rename / delete / export / import），结构完全相同，仅 API 路径、DOM 选择器 ID 和列表变量名不同。

```javascript
// loadCCConfigs vs loadCXConfigs — 14 行中 14 行相同，只有 3 个标识符不同
async function loadCCConfigs() {
  const data = await api('/api/claude-code/configs', 'GET');      // ← 差异
  ccConfigList = data.configs || [];                               // ← 差异
  const sel = document.getElementById('cc-config-select');         // ← 差异
  // ... 后面 11 行完全相同
}
```

> **建议（P0——强烈建议）：**
> 1. **Server 端**：提取通用 `ConfigStore` 工厂函数：
>    ```javascript
>    function createConfigStore(filePath, sanitizeFn) {
>      return {
>        read() { ... },
>        write(data) { ... },
>        list() { ... },
>        create(name, config) { ... },
>        apply(id) { ... },
>        delete(id) { ... },
>        rename(id, name) { ... },
>        export(id, stripKey) { ... }
>      };
>    }
>    const ccStore = createConfigStore('cc-configs.json', sanitizeCC);
>    const cxStore = createConfigStore('codex-cli-configs.json', sanitizeCX);
>    const cxdStore = createConfigStore('codex-desktop-configs.json', sanitizeCXD);
>    ```
>    **预期节省：~350 行 server.js**
> 2. **Server 端**：提取路由注册辅助函数：
>    ```javascript
>    function registerCrudRoutes(prefix, store) { ... }
>    ```
>    **预期节省：~150 行 server.js**
> 3. **Index.html 端**：参数化 UI 管理函数：
>    ```javascript
>    function createConfigManager(productKey, apiPrefix, selectId) { ... }
>    ```
>    **预期节省：~450 行 index.html**
> 4. **总计预期收益：消除 ~950 行重复代码（占当前代码量约 18%）**

### 2.2 HTML 工具栏模板重复

4 个产品配置 Tab 各有一个几乎相同的预设管理工具栏（选择器 + 新建/重命名/删除/导出/导入按钮），每个 12 行 HTML。

> **优先级：中** | **建议：** 提取为 JS 生成的组件函数 `renderConfigToolbar(productKey, buttons)`，减少 44 行 HTML。

## 四、代码层面的交互隐患

### 4.1 定时器泄漏（内存泄漏）

> **优先级：高**

```javascript
// index.html 第 2520-2521 行
setInterval(pollLogs, 2000);          // 每 2 秒轮询日志
setInterval(refreshProcIndicator, 8000); // 每 8 秒轮询进程状态
```

**问题：** 这两个 `setInterval` **永远不被清除**。整个项目中没有任何 `clearInterval` 调用。即使用户切到其他浏览器 Tab、最小化窗口，这两个定时器持续运行：
- `pollLogs` 虽然面板关闭时提前返回（`if (!logPanelOpen) return`），但**每 2 秒仍执行一次函数调用**
- `refreshProcIndicator` 无条件执行 HTTP 请求查询进程状态

**缺少：** `visibilitychange` 事件监听器（页面不可见时暂停轮询）、`beforeunload` 清理逻辑。

> **修复建议：**
> ```javascript
> const pollTimers = [];
> pollTimers.push(setInterval(pollLogs, 2000));
> pollTimers.push(setInterval(refreshProcIndicator, 8000));
> document.addEventListener('visibilitychange', () => {
>   if (document.hidden) pollTimers.forEach(clearInterval);
>   else { /* 重新启动定时器 */ }
> });
> ```

### 4.2 冗余状态加载（性能退化）

> **优先级：高**

`loadState()` 是前端的状态加载入口函数。每调用一次触发 **7 个 API 请求** 的级联：

```
loadState()
  → api('/api/state')         [1]
  → loadCCConfigs()           [2]  /api/claude-code/configs
  → loadCDConfigs()           [3]  /api/claude-desktop/configs
  → loadCXConfigs()           [4]  /api/codex-cli/config/list
  → loadCXDConfigs()          [5]  /api/codex-desktop/config/list
  → loadAppPaths()            [6]  /api/paths
  → proxyRefreshStatus()      [7]  /api/proxy/status
  + renderStatusGrid() + renderUpstreamTables() + renderMappingForms()
```

**`loadState()` 被调用了 29 次**——分布在每个保存、创建、删除、重命名、导入、回滚操作之后。这意味着：
- 修改 Claude Code 配置 → 重新加载 Codex Desktop 配置（完全无关）
- 修改 Gateway 配置 → 重新加载 App Paths（完全无关）
- 每点一次「保存」→ 7 个 HTTP 请求 + 全量 DOM 重建

> **修复建议：**
> 1. 按需加载：`saveClaudeCode()` 后只调用 `loadCCConfigs()`，不触发 `loadState()`
> 2. Tab 懒加载：仅在首次切换到某 Tab 时加载对应数据，切换回来时使用缓存
> 3. 脏标记：保存成功后只更新当前产品的状态切片，不重新拉取所有产品

### 4.3 DOM 抖动（布局颠簸）

> **优先级：中**

**日志面板（2 秒一次 500 条目全量重绘）——最严重：**
```javascript
// index.html 第 2512 行
b.innerHTML = logs.slice().reverse().map(l => `<div class="log-line">...</div>`).join('');
```
调试日志面板在打开时每 2 秒执行一次**全量 DOM 重建**：销毁 500 个旧 `<div>`，创建 500 个新 `<div>`，触发浏览器布局重计算。这是页面运行时最大的性能消耗源。

**模型列表无上限渲染：**
```javascript
// index.html 第 1230 行
container.innerHTML = data.models.map(m => `<div>...</div>`).join('');
```
如果上游返回 1000+ 模型（大厂 API 常见），一次性创建 1000 个 DOM 节点。无分页、无虚拟滚动。

> **修复建议：**
> 1. 日志面板：使用增量更新而非全量替换——只插入新条目，超出 500 条时移除最旧条目
> 2. 模型列表：添加分页（50 条/页）或虚拟滚动
> 3. 使用 `DocumentFragment` 或 `requestAnimationFrame` 批量 DOM 写入

### 4.4 错误处理缺失（运行时崩溃风险）

> **优先级：高**

**【关键】`api()` 函数没有网络错误处理：**
```javascript
// index.html 第 946-951 行
async function api(path, method, body) {
  const opts = { method, headers: { 'Content-Type': 'application/json' } };
  if (body) opts.body = JSON.stringify(body);
  const res = await fetch(path, opts);  // ← 网络错误 = 未捕获 Promise rejection
  return res.json();                    // ← 非 JSON 响应 = SyntaxError — 同样未捕获
}
```

65 个调用 `api()` 的地方依赖各自的 try/catch，但以下关键调用**没有错误处理**：
- `saveProxy()`（第 1831 行）—— `await api('/api/save', ...)` 没有 try/catch
- `saveClaudeCode()`（第 1527 行）—— 主保存调用没有 try/catch

**Server 端无全局异常处理：**
```javascript
// server.js 中完全不存在以下代码
process.on('uncaughtException', ...);
process.on('unhandledRejection', ...);
```
一个异步路由处理器的抛出异常会导致**整个 Node.js 进程崩溃**，包括正在运行的网关（端口 9877）—— 所有正在中转的请求同时断开。

> **修复建议：**
> 1. `api()` 函数内增加 `try/catch` + `res.ok` 检查，返回 `{ error, status }` 结构化对象
> 2. Server 端添加全局异常处理器，至少记录错误并优雅返回 500，不崩溃进程
> 3. 消除所有静默吞错的 empty catch 块（至少 `console.error` 输出）

### 4.5 XSS 风险

> **优先级：中**

**`esc()` 函数未转义单引号：**
```javascript
// index.html 第 1442 行
function esc(s) { return String(s).replace(/&/g,'&amp;').replace(/</g,'&lt;')
                                   .replace(/>/g,'&gt;').replace(/"/g,'&quot;'); }
//                                                                      ↑ 缺失 ' 转义
```

这导致在单引号分隔的属性中存在注入风险：
```html
<!-- 如果模型 ID 包含单引号，可逃逸出 onclick 处理函数 -->
onclick="copyToClipboard('${esc(String(m.id))}')"
```

`esc()` 用了正确的白名单思路，但遗漏了单引号。如果模型 API 返回的 ID 包含 `'); alert(1);//`，将造成 XSS。

**未转义直接渲染用户数据：**
```javascript
// index.html 第 1060 行
grid.innerHTML = items.map(i => `... ${i.url} ...`).join('');
//                                    ↑ i.url 完全没有经过 esc() 处理
```

> **修复建议：**
> 1. `esc()` 补上 `'` → `&#39;` 或 `&apos;`
> 2. 所有 `innerHTML` 赋值的数据路径统一经过 `esc()`
> 3. 长期：考虑用 `textContent` 替代 `innerHTML`，仅在需要富文本时使用后者

### 4.6 Server 端 Content-Disposition 头注入

> **优先级：中**

```javascript
// server.js 第 2538 行
const fname = 'claude-code-' + (payload.name || 'config').replace(/[^\w.-]+/g, '_') + suffix + '.json';
res.setHeader('Content-Disposition', `attachment; filename="${fname}"`);
```

如果用户在配置预设名称中输入 `"`，虽然会被 `replace(/[^\w.-]+/g, '_')` 替换为 `_`（因为 `"` 不匹配 `\w`），此处实际**安全**。但如果未来修改了 sanitize 逻辑放宽了字符集，`"` 可以破坏 HTTP 头语法。建议使用 `encodeURIComponent` 包裹 filename 值。

### 4.7 服务端资源清理

> **优先级：中**

网关转发处理器中，`upstreamRes` 的事件监听器在 `'end'` 和 `'error'` 事件中会被清理（`server.js:1741-1761`），SSE 心跳定时器也在流关闭时正确清除。**这部分实现规范。**

但全局 `server.on('error', ...)` 只记录了日志（`server.js:2766`），没有优雅关闭逻辑（清理进行中的连接、写入持久化状态、flush 日志缓冲区）。
