## 一、信息架构与页面层级（IA Review）

### 1.1 「虚假导航」假设验证

**结论：假设部分成立。** 顶部导航不是"完全虚假"——Tab 切换是真实有效的（`switchTab()` 第 921 行确实切换了 `.tab-panel` 的显示/隐藏）。但存在三类信息架构问题：

#### 问题 1A：设置控件混入导航区域，造成"伪 Tab"错觉

> **优先级：高**

header 栏从左到右排列了：
1. Logo（`<h1>`）
2. 7 个真正的 Tab（`<nav class="tabs">`）
3. 主题选择器（`<select>`）
4. **边栏布局选择器**（`<select>`）
5. 进程状态指示器

```
⚡ RelayManager | [总览] [Claude Code] [Claude Desktop] [Codex CLI] [Codex Desktop] [模型映射] [中转网关] | 主题[▼] 边栏[▼] ●检查中…
```

**问题**：主题和边栏选择器在视觉上与 Tab 处于同一水平线，且使用相同的暗色背景、圆角和字体大小。用户初看会以为"经典侧栏"、"紧凑侧栏"、"侧边栏导航"是可切换的页面/视图，但它们实际是**全局布局设置**，不改变当前显示的面板内容。

"边栏"标签本身也有歧义——前两个选项（经典侧栏、紧凑侧栏）并不改变导航结构，只是调整了 Tab 的 padding/font-size/max-width，只有第三个选项（侧边栏导航）真正把顶部 Tab 变成了左侧导航。用户选择"经典侧栏"时期望看到侧边栏，实际什么都没变。

> **建议**：
> 1. 将主题和边栏选择器移到页面底部或单独的"设置"面板，与导航区域视觉分离
> 2. "边栏"下拉重命名为"布局密度"或拆分为两个独立开关：**导航位置**（顶部/左侧）和**界面密度**（标准/紧凑）
> 3. 至少用竖线分隔符（`|`）或背景色差异将导航 Tab 和设置控件明显区分

#### 问题 1B：7 个平级 Tab 违反信息层级

> **优先级：高**

当前 7 个 Tab 在语义上属于三个不同的信息层级，却被强制平级排列：

| Tab | 实际语义 | 应属层级 |
|---|---|---|
| 总览 | 仪表板/首页 | L1 — 入口 |
| Claude Code / Claude Desktop / Codex CLI / Codex Desktop | 产品配置 | L2 — 核心任务 |
| 模型映射 | 工具/辅助 | L3 — 辅助操作 |
| 中转网关 | 基础设施 | L2 — 核心任务（但独立于产品） |

**具体问题**：
- "模型映射" Tab 的内容与各产品 Tab 强耦合（它为 4 个产品分别提供映射表单），但被提升为独立 Tab，导致用户需要在"产品 Tab"和"模型映射 Tab"之间来回跳转
- "总览" Tab 承载了 7 张卡片（一键同步、配置状态、连通性检测、代理控制、重启客户端、应用路径、开机自启），信息密度极高，但没有任何子导航或折叠机制

> **建议的重构方案**：
> ```
> 侧边栏导航结构：
> ├── 🏠 总览（仪表板）
> ├── 📦 产品配置
> │   ├── Claude Code CLI
> │   ├── Claude Desktop 3P
> │   ├── Codex CLI
> │   └── Codex Desktop
> ├── 🛡 中转网关
> └── ⚙ 工具
>     ├── 模型映射
>     ├── 代理控制
>     └── 应用路径 & 开机自启
> ```
> 实施代价：约 40 行 CSS（利用现有 `data-sidebar-layout="sidebar"` 的 grid 布局）+ 30 行 HTML 结构调整。

#### 问题 1C：`models` Tab 的内容越界

> **优先级：中**

`models` Tab 名为"模型映射"，但实际包含 4 个功能块：
1. **从上游拉取模型列表**（通用工具）
2. **Claude Code 模型映射**（表单编辑）
3. **Claude Desktop 模型映射**（仅一个跳转按钮，指向 Claude Desktop Tab）
4. **Codex CLI 模型映射**（表单编辑）
5. **Codex Desktop 模型映射**（JSON 编辑）

第 3 块（Claude Desktop）是一个**纯跳转按钮**，直接告诉用户"去 Claude Desktop 标签页编辑"。这暴露了设计上的自我矛盾——既然模型映射是各产品配置的一部分，为何不直接放在各产品 Tab 内？

> **建议**：将模型映射功能内聚到各产品 Tab 中（Codex CLI 的映射表单已经在产品 Tab 中有 `<details>` 高级设置区域，Claude Code 的 6 个模型字段也在产品 Tab 中），删除独立 `models` Tab，将"从上游拉取模型列表"功能移到中转网关 Tab 或总览的连通性检测卡片中。

## 二、功能重复性检测（DRY）

### 2.1 「一键同步」vs「应用到产品」重复假设验证

**结论：假设完全成立。** 这两个功能共享约 70% 的核心逻辑。

> **优先级：高**

#### 功能 A：`doSyncAll()`（总览 Tab，第 1490 行）

```
用户操作流程：
1. 在「一键同步所有产品」卡片填写 Base URL + API Key
2. 勾选目标产品复选框
3. 点击「应用到所有产品」
4. 后端 /api/save 接收 syncAll 字段，统一写入所有勾选产品的配置
```

#### 功能 B：`applyRelayToSelectedProducts()`（连通性检测卡片，第 1721 行）

```
用户操作流程：
1. 在「中转站连通性检测」卡片填写 Base URL + API Key
2. （可选）先点「测试连接」验证连通性
3. 勾选「应用到产品」复选框
4. 点击「应用到所选产品并保存」
5. 对每个勾选的产品：
   - Claude Code / Codex CLI → 弹 prompt 命名，创建预设
   - Claude Desktop / Codex Desktop → 直接保存
```

#### 重叠分析

| 步骤 | `doSyncAll` | `applyRelayToSelectedProducts` |
|---|---|---|
| 填写 Base URL + Key | ✅ | ✅ |
| 勾选目标产品 | ✅ | ✅ |
| 回填表单字段 | ❌（后端直接写入） | ✅（前端 setVal 回填各产品表单） |
| 调用保存逻辑 | ✅（统一 syncAll） | ✅（分产品调用） |
| 创建预设 | ❌ | ✅（CC/CX 弹 prompt） |
| 重启客户端 | ❌ | ✅（可选） |

两者的**用户意图完全相同**："把一组 Base URL + Key 应用到多个产品"。但它们是两套独立的 UI、两套独立的代码路径，而且：
- **位置相邻**：都在总览 Tab，垂直距离不到 200px
- **字段重复**：`sync-base-url` / `sync-api-key` 与 `test-base-url` / `test-api-key` 是完全相同的输入需求
- **复选框重复**：`sync-cc/sync-cd/sync-cx/sync-cxd` 与 `test-apply-target` 中的复选框语义完全相同

> **建议**：合并为一个统一的「应用配置到产品」工作流：
> ```
> [Base URL 输入框] [API Key 输入框]
> [测试连接] [获取模型列表]
> ☑ Claude Code  ☑ Claude Desktop  ☑ Codex CLI  ☑ Codex Desktop
> ☐ 应用后自动重启客户端
> [⚡ 应用到所选产品]
> ```
> 合并后删除 `doSyncAll()` 和独立的"一键同步"卡片，保留连通性检测卡片的完整功能（因为它包含了测试连接能力，是功能超集）。预期减少 ~50 行 JS + ~30 行 HTML。

## 四、UI/UX 交互体验

### 4.1 按钮风格统一性

> **优先级：高**

当前定义了 6 种按钮样式：

| 类名 | 语义 | 使用场景 |
|---|---|---|
| `.btn-primary` | 主要操作 | 保存、应用 |
| `.btn-green` | 成功/启动 | 启动代理、应用到 CD |
| `.btn-red` | 危险/删除 | 删除配置、停止代理 |
| `.btn-yellow` | 警告/重启 | 重启客户端 |
| `.btn-outline` | 次要操作 | 测试连接、添加、导出 |
| `.btn-sm` / `.btn-lg` | 尺寸变体 | 叠加在其他样式上 |

**发现的不统一问题**：

**A) 删除按钮混用了两种风格：**

```html
<!-- 预设管理工具栏：btn-red（正确——这是破坏性操作） -->
<button class="btn btn-red btn-sm" onclick="deleteCCConfigUI()">✕ 删除</button>

<!-- 上游表格行内删除：btn-red（正确） -->
<button class="btn btn-red btn-sm" onclick="deleteUpstream(${i})">✕</button>

<!-- 但网关模型映射行内删除：也是 btn-red（正确） -->
<button class="btn btn-red btn-sm" onclick="removeGWRoute(${i})">✕</button>
```
✅ 删除按钮风格一致。

**B) 保存按钮混用了两种强调级别：**

```html
<!-- 大多数产品：btn-primary -->
<button class="btn btn-primary" onclick="saveClaudeCode()">💾 保存 Claude Code</button>

<!-- 网关「应用到 Claude Desktop」：btn-green（语义混淆——这不是"启动"操作） -->
<button class="btn btn-green" onclick="applyGatewayToClaudeDesktop()">⚡ 应用到 Claude Desktop</button>
```
❌ "应用到 Claude Desktop" 是保存/应用操作，不是启动操作，应使用 `btn-primary`。绿色在此处语义不匹配。

**C) 代理控制按钮语义混乱：**

```html
<button class="btn btn-green btn-sm" onclick="proxyStart()">▶ 启动代理</button>
<button class="btn btn-red btn-sm" onclick="proxyStop()">■ 停止代理</button>
<button class="btn btn-yellow btn-sm" onclick="restartApp('proxy')">↻ 重启代理</button>
```
✅ 启动/停止/重启三者语义正确。
❌ 但 `restartApp('proxy')` 调用的是通用重启函数，使用的是 `btn-yellow`——而 `restartApp('claude-desktop')` 也在用 `btn-yellow`，语义一致，没问题。

**D) "测试连接"按钮风格不一：**

```html
<!-- 总览 Tab：btn-outline -->
<button class="btn btn-outline btn-sm" onclick="testRelayConnection()">🔌 测试连接</button>

<!-- Claude Desktop Tab：btn-outline（一致 ✅） -->
<button class="btn btn-outline btn-sm" onclick="testCDConnection()">🔌 测试连接</button>

<!-- Codex CLI Tab：btn-outline（一致 ✅） -->
<button class="btn btn-outline btn-sm" onclick="testCXConnection()">🔌 测试连接</button>

<!-- Claude Code Tab：btn-outline（一致 ✅） -->
<button class="btn btn-outline btn-sm" onclick="testCCMessage()">🧪 真实对话自测</button>
```
✅ 所有连接测试按钮一致使用 `btn-outline`。

**E) 缺失的按钮状态**：当前没有 `btn:disabled` 样式（只有 `.btn:active`）。当 `showLoading()` 禁用按钮时，按钮外观没有视觉变化（缺少 `opacity: 0.5` 或 `cursor: not-allowed`），用户可能以为按钮仍然可点击。

> **建议**：
> 1. 将 `applyGatewayToClaudeDesktop` 的按钮从 `btn-green` 改为 `btn-primary`
> 2. 添加 `.btn:disabled { opacity: 0.5; cursor: not-allowed; }` 样式
> 3. 抽取按钮 CSS 变量：`--btn-primary-bg`、`--btn-danger-bg` 等，确保主题切换时所有按钮一致变色

### 4.2 操作反馈机制（Toast / 加载状态 / 提示）

> **优先级：高**

#### 覆盖情况分析

| 操作 | Toast 通知 | 加载状态 | 成功状态 | 失败状态 |
|---|---|---|---|---|
| `doSyncAll()` | ✅ | ✅ (`showLoading`) | ✅ | ✅ |
| `saveClaudeCode()` | ✅ | ❌ | ✅ | ❌（无 catch） |
| `saveClaudeDesktop()` | ✅ | ❌ | ✅ | ❌（无 catch） |
| `saveCodexCli()` | ✅ | ❌ | ✅ | ❌（无 catch） |
| `saveCodexDesktopUpstream()` | ✅ | ❌ | ✅ | ❌（无 catch） |
| `saveGateway()` | ✅ | ❌ | ✅ | ✅ |
| `restartApp()` | ✅ | ✅ | ✅ | ✅ |
| `applyRelayToSelectedProducts()` | ✅ | ❌ | ✅ | ❌ |
| 预设创建/删除/重命名 | ✅ | ❌ | ✅ | ✅ |
| 连接测试 (所有) | ❌ | ✅ (loading spinner) | ✅ (inline) | ✅ (inline) |
| 主题/边栏切换 | ❌ | N/A | ✅ (即时) | ❌ |

**发现的缺失**：

**A) `saveClaudeCode()` 没有错误 Toast（第 1509-1534 行）：**
```javascript
async function saveClaudeCode() {
  // ...
  await api('/api/save', 'POST', data);  // ← 没有 try/catch！
  // 若此行抛出异常，用户不会收到任何错误提示
  toast('Claude Code 配置已保存...');  // ← 这行不会执行
}
```
如果 `/api/save` 返回非 JSON 响应或网络错误，`api()` 函数会抛出 `SyntaxError`，但此函数没有 try/catch，Promise rejection 未被处理。**用户点击保存按钮后，如果失败，界面完全静默——按钮没有反馈，Toast 不出现，状态不改变。**

同样的问题存在于 `saveClaudeDesktop()`（第 1568 行）、`saveCodexCli()`（第 1636 行）、`saveCodexDesktopUpstream()`（第 1695 行）。

**B) 连接测试使用 inline status 而非 Toast：**
连接测试的结果显示在按钮旁边的 `<span>` 中，而不是 Toast。这在视觉上不一致——有时错误出现在按钮旁边，有时出现在右上角。

> **建议**：
> 1. 所有 `save*` 函数统一加上 try/catch + Toast 错误提示（这是**最高优先级的修复**）
> 2. 为保存操作添加统一的加载状态（按钮文字变为"保存中..."并禁用按钮）

## 五、代码质量

### 5.1 JavaScript 重复逻辑与内存泄漏

> **优先级：高**

#### 泄漏 A：两个永不清理的 setInterval（第 2520-2521 行）

```javascript
setInterval(function(){ if(logPanelOpen) pollLogs(); }, 2000);
setInterval(refreshProcIndicator, 8000);
```

这两个定时器：
- 在页面加载时创建
- **永远不被 `clearInterval` 清除**
- 即使浏览器 Tab 不可见也持续运行
- `pollLogs` 虽然面板关闭时提前返回，但函数调用本身仍然执行（约每 2 秒一次的 fetch 请求检查 `if (!logPanelOpen) return`）

**修复**：
```javascript
const logTimer = setInterval(() => { if (logPanelOpen) pollLogs(); }, 2000);
const procTimer = setInterval(refreshProcIndicator, 8000);
document.addEventListener('visibilitychange', () => {
  if (document.hidden) {
    clearInterval(logTimer);
    clearInterval(procTimer);
  }
});
```

#### 泄漏 B：`api()` 函数无错误处理（第 946-951 行）

```javascript
async function api(path, method, body) {
  const opts = { method, headers: { 'Content-Type': 'application/json' } };
  if (body) opts.body = JSON.stringify(body);
  const res = await fetch(path, opts);  // ← 网络错误 → 未捕获 rejection
  return res.json();                    // ← 非 JSON 响应 → SyntaxError
}
```

此函数被 65 处调用。以下调用点**没有 try/catch 包裹**（确认自代码遍历）：

- `saveProxy()` 第 1838 行 — `await api('/api/save', ...)`
- `saveClaudeDesktop()` 第 1568 行 — `await api('/api/save', ...)`
- `saveCodexDesktopUpstream()` 第 1695 行 — `await api('/api/save', ...)`
- `saveMappingCC()` 第 1348 行 — `await api('/api/save', ...)`
- `saveMappingCXD()` 第 1414 行 — `await api('/api/save', ...)`

**修复**：
```javascript
async function api(path, method, body) {
  try {
    const opts = { method, headers: { 'Content-Type': 'application/json' } };
    if (body) opts.body = JSON.stringify(body);
    const res = await fetch(path, opts);
    if (!res.ok) {
      const text = await res.text();
      let msg = text;
      try { msg = JSON.parse(text).error || text; } catch (e) {}
      return { error: msg, status: res.status };
    }
    return res.json();
  } catch (e) {
    return { error: e.message, status: 0 };
  }
}
```
然后更新所有调用点检查 `res.error`。

#### 泄漏 C：`esc()` 函数缺单引号转义（第 1442 行）

```javascript
function esc(s) { return String(s).replace(/&/g,'&amp;').replace(/</g,'&lt;')
                                  .replace(/>/g,'&gt;').replace(/"/g,'&quot;'); }
//                                                                     ↑ 缺 ' 的转义
```

影响所有在单引号属性中使用 `esc()` 的地方：
```javascript
// 第 1235 行 — 单引号分隔的 onclick
`onclick="copyToClipboard('${esc(String(m.id))}')"`

// 第 1902 行 — 同样的模式
`onclick="copyToClipboard('${esc(String(m.id))}')"`

// 第 2516 行 — 历史回滚
`onclick="doRollback('${esc(it.filename)}')"`
```

如果模型 ID 包含单引号（罕见但可能），XSS 攻击面打开。

**修复**：`esc()` 补上 `.replace(/'/g, '&#39;')`

#### 泄漏 D：Server 端无全局异常处理

```bash
$ grep -n "uncaughtException\|unhandledRejection" server.js
# 无结果 —— 完全没有注册全局异常处理器
```

网关（端口 9877）是一个生产服务。一个异步路由处理器的未捕获异常会**崩溃整个 Node.js 进程**，同时中断网关转发和 HTTP API。

**修复**：
```javascript
process.on('uncaughtException', (err) => {
  console.error('[FATAL] Uncaught exception:', err.message);
  debugLog('error', 'uncaught_exception', err.message);
  // 不立即 process.exit(1)，让现有连接有机会完成
});
process.on('unhandledRejection', (reason) => {
  console.error('[FATAL] Unhandled rejection:', reason);
  debugLog('error', 'unhandled_rejection', String(reason));
});
```
